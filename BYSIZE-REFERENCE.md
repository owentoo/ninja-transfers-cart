# By-Size Product — Cart Edit Size: Source of Truth

> This document covers every piece of data, logic, and configuration used by the
> Cart Edit Size popup for DTF / UV "by-size" products on transferss.myshopify.com.

---

## 1. What Is a "By-Size" Product?

A by-size product is one where the customer enters custom width × height dimensions
instead of picking a fixed variant. The variant selected at add-to-cart maps to a
square-inch price tier (e.g. "1–9 sq in", "10–24 sq in"). Product types that qualify:

- Custom Transfers (DTF)
- UV DTF
- Spangles
- Puff Transfers
- PermaSticker

Detection in Liquid: `lineItem.product.type contains 'UV'` or `'PermaSticker'` flags
`_ap_is_uv = true` in `arrange-properties.liquid`. All others are treated as standard DTF.

## 4. DPI Warning Logic

### Source
Original logic lives in `assets/pdp_transferBySize.js` → `calculateDPILineItem()` and
`calculateDPI()`. The cart popup replicates both conditions exactly.

### Threshold variable
```
PDP:  var grt_dpi_warning = {{ block.settings.dpiwarning_input }}  (default: 72)
Cart: var NT_MIN_DPI = 72;  (hardcoded in cart-edit-size.js)
```

> ⚠️ **ACTION REQUIRED — verify before relying on perfect parity:**
> If the `dpiwarning_input` section setting has been changed from `72` in the
> Shopify theme editor (bySize section → fileupload block → "DPI Warning number"),
> `NT_MIN_DPI` in `cart-edit-size.js` must be updated to match.
> Mismatched values cause the warning to fire at different sizes than the PDP.

### Two conditions that trigger the warning

**Condition 1 — Image too small in absolute terms (always warn):**
```
( imgPixelWidth × imgPixelHeight ) / NT_MIN_DPI < 288
```
If true, the image has fewer than ~20,736 total pixels. No print size is safe.
Warning shows regardless of selected size. Only fix for the customer is a better upload.

Equivalent PDP check: `makeSqInchesForCustom < 288` where
`makeSqInchesForCustom = (uploadedFileWidth × uploadedFileHeight) / grt_min_dpi`

**Condition 2 — Selected size exceeds image resolution:**
```
selectedSqIn > ( imgPixelWidth / NT_MIN_DPI ) × ( imgPixelHeight / NT_MIN_DPI )
```
Selected print area is larger than what the image can cover at minimum DPI.

Equivalent PDP check (popular path): `makeSqInchesForPopular > recommendedSqInches`
where `recommendedSqInches = (uploadedFileWidth / grt_min_dpi) × (uploadedFileHeight / grt_min_dpi)`

### File types that trigger the check
Only raster files: `.png`, `.jpg`, `.jpeg`
Vector files (`.ai`, `.eps`, `.pdf`, `.svg`) are skipped — they scale infinitely.

### How pixel dimensions are obtained (runtime, no PDP changes)
```
1. Fast path:  fetch( imgixUrl + '?fm=json' )
               → parses { PixelWidth, PixelHeight } from imgix metadata endpoint
               → near-instant, no full image download

2. Fallback:   new Image() loaded from raw URL (query params stripped)
               → reads naturalWidth / naturalHeight on load
               → always works; slower for large files

3. Failure:    if both fail, imgPx stays null → warning never shows (silent)
```

### Immediate show on modal open (synchronous)
Before the async dimension load completes, the modal checks for `DPI Warning: Yes`
in the stored cart item properties. If present, the warning is shown immediately
(zero network round-trip). The async load then recalculates and may hide it if the
user had since changed to a smaller size and the property is stale.

### Visual treatment when warning is active
- Warning text appears below size inputs: `#fea333`, `font-size:12px`, `font-weight:500`
- Custom Size tab: both width + height input container borders → `#fea333`, `2.5px`
- Popular Size tab: dropdown border → `#fea333`, `2.5px`
- On warning clear: all borders reset to `#ddd`, `1.5px`

## 5. Variant / Price Tier Lookup

Variants use range-title format: `"1-9"`, `"10-24"`, `"25-49"`, etc.
Square inches = `Math.max(1, Math.round(width × height))` — minimum 1 sq in.

Lookup order in `findVariantForSqIn()`:
1. Range-title match, available variants only
2. Range-title match, ignore availability
3. Fallback: highest `max` number in any variant title that is ≥ sqIn

Anything below 1 sq in (e.g. 0.25" × 0.25") rounds up to 1 sq in for pricing.

## 6. Discount Tiers (`_discount_input` property)

Stored as a triple-URL-encoded string. Format after full decode:
```
minQty_discountPct-minQty_discountPct-…#minQty_discountPct-…
```
Each `#`-separated segment is one tier group. Within a group, `-` separates entries.
Each entry: `{minQty}_{pct}` where pct is the integer discount percentage.

Effective total quantity for tier lookup:
```
effectiveTotal = ( totalDtfQty - itemQty ) + editQty
```
`totalDtfQty` = sum of quantities across all cart items that have an upload property.
This ensures cross-item cumulative discounts are respected when editing a single item.

## 7. Smart Sizing (Popular Size tab)

When the user picks a popular size, the popup fits the item's original aspect ratio
(from stored Width × Height) into the chosen bounding box rather than using the exact
popular size dimensions. This avoids distortion.

Logic: if `aspectRatio ≥ bw/bh`, lock to bounding width; else lock to bounding height.
Result is clamped to `[minDim, maxDim]` from product metafields
(`custom.minimum_measurement` default 0.25, `custom.maximum_measurement` default 22).

"Smart Sizing Applied" label (green, with ⓘ tooltip) shows when the fitted dimensions
differ from the raw popular size by more than 0.01".

| `data-total-dtf-qty` | Sum of quantities across all cart items with an upload property |

| `data-min-dim` | `product.metafields.custom.minimum_measurement` (default 0.25) |
| `data-max-dim` | `product.metafields.custom.maximum_measurement` (default 22) |

## 10. Update Flow (confirm button)

1. Read `w` and `h` from `#nt-edit-w` / `#nt-edit-h` (popular tab pre-fills these)
2. Compute `sqIn = Math.max(1, Math.round(w × h))`
3. Look up variant → `variantId`
4. Clone all existing properties; overwrite `Width` and `Height` (preserve original key casing)
5. `POST /cart/change.js` with `{ id: lineKey, quantity: 0 }` — removes old line
6. `POST /cart/add.js` with new `{ variantId, quantity: editQty, properties }` — adds updated line
7. `window.location.reload()` on success

Note: remove-then-add is used (not `/cart/change.js` quantity + properties) because
Shopify's change endpoint cannot update line item properties in one call.
